=== ノートの Export / Import

ノートの Export / Import は、通常のファイル upload / download とは区別する。

upload / download は、個別の Content resource に対応するファイル操作である。
一方、Export / Import は、ノート本体、ノート内の managed resource、及び upload 時又は export 時に生成された thumbnail や preview PDF などを、交換可能な package として扱う処理である。

==== 改定の目的

同じ実行環境内で Export / Import した場合、note JSON だけを移動しても `node.resource.storage.files[]` が参照する実体ファイルは既存ディレクトリに残るため、見かけ上は復元できたように見える。

しかし、別ユーザ、別サーバ、又は別環境へ Import する場合、`resource.storage.files[]` が参照する実体ファイルが zip package に含まれていなければ、Import 後に Content resource を開けない。

このため、Export では、note JSON だけでなく、`resource.storage.files[]` に対応する実体ファイルを package に同梱する。
Import では、同梱ファイルを import 実行ユーザの保存領域へ復元し、note JSON 内の参照を import 実行ユーザの環境で参照可能な状態に補正する。

==== 用語

`note JSON`::
  ノート本体を表す JSON ファイル。

`export manifest`::
  package に含まれる note JSON、resource file、変換結果、SHA256 などを記録する `export-manifest.json`。

`managed resource`::
  WuWei の管理下にある実体ファイルを持つ resource。
  `resource.storage.managed === true` 又は `resource.storage.files[]` に保存先が定義されている Content resource を指す。

`resource file`::
  `resource.storage.files[]` に定義された role 別ファイル。
  例: `original`, `preview`, `thumbnail`, `pdf-preview`。

`logical path`::
  ユーザ UUID を含まない保存用 path。
  例: `YYYY/MM/DD/file_uuid/filename`。

`physical relative path`::
  サーバルートからの相対 path。
  例: `data/{user_uuid}/upload/YYYY/MM/DD/file_uuid/filename`。

`package staging area`::
  Export / Import の一時作業領域。
  例: `data/public/YYYY/MM/DD/package_uuid/`。

==== 基本方針

Export では、次の情報を zip package にまとめる。

* note JSON
* export manifest
* managed resource の role 別ファイル
* upload resource の original file
* upload resource から生成された preview file
* upload resource から生成された thumbnail file
* Office 文書の original file に代えて同梱する PDF 変換ファイル
* その他 `resource.storage.files[]` に定義された role 別ファイル

remote resource は、原則として実体ファイルを package に含めない。
remote resource は、`resource.uri` 又は `resource.canonicalUri` に記録された URL をそのまま保持する。

Import では、zip package を一時的に次の public 作業領域へ展開する。

[source,text]
----
data/public/YYYY/MM/DD/package_uuid/
----

この public 領域は、Export / Import の一時作業領域であり、Import 後の正式な保存先ではない。

Import 後、同梱された managed resource の実体ファイルは、import 実行ユーザの保存領域へコピーする。

[source,text]
----
data/{import_user_uuid}/upload/YYYY/MM/DD/file_uuid/
----

Import 後の note JSON は、import 実行ユーザの note 領域に保存する。

[source,text]
----
data/{import_user_uuid}/note/YYYY/MM/DD/note_uuid/note.json
----

==== ディレクトリ構成の前提

ユーザ別の管理対象ファイルは、次の構成で保存する。

[source,text]
----
data/
  {user_uuid}/
    note/
      YYYY/
        MM/
          DD/
            note_uuid/
              note.json
    upload/
      YYYY/
        MM/
          DD/
            file_uuid/
              original または PDF 変換ファイル
              preview.pdf
              thumbnail.jpg
              manifest.json
    content/        // v0 / legacy content を許容する場合
    thumbnail/      // v0 / legacy thumbnail を許容する場合
  public/
    YYYY/
      MM/
        DD/
          package_uuid/
            export-manifest.json
            note/
            resources/
----

v2 の標準保存先は `upload` とする。
ただし、v0 / legacy ノートには `content` や `thumbnail` を参照するものがあるため、Export では `resource.storage.files[].area` を見て、`upload`, `content`, `thumbnail`, `resource`, `note` の実体ファイルを読み取れるようにする。

==== UUID の扱い

SVG 描画対象となる `node.id`、`link.id`、`group.id` などは、SVG / DOM 上の識別子として使用されるため、次の形式とする。

[source,javascript]
----
"_" + uuid.v4()
----

一方、新規 v2 で作成する `note_uuid`、`file_uuid`、`package_uuid` は、論理識別子及びディレクトリ名として使用するため、先頭に `_` を付けず、次の形式とする。

[source,javascript]
----
uuid.v4()
----

ただし、v0 時代のノートでは `note_id` 又は `note_uuid` が `_` で始まる値を持つことがある。
Import は、v0 ノートの `_` 付き `note_id` / `note_uuid` を legacy note identifier として許容する。
この場合も、`node.id` / `link.id` の `_` 付き UUID とは意味を区別する。

例:

[source,text]
----
node.id:
  _6c799e9d-1b5b-4a6e-bf4d-d61f7c9fd8be

v2 note_uuid:
  5c1f594a-315b-4bab-9075-62b1bebed7eb

v0 note_uuid:
  _f855cedd-898a-469d-87eb-ebd3e672c42f

file_uuid:
  8ed59104-779e-4907-9978-6f0f1acb8cf5
----

==== note_uuid と note_key

`note_uuid` は、エクスポート対象ノートの UUID とする。

`note_key` は、ノートの論理保存位置を示す user-independent な path とする。

[source,javascript]
----
note_uuid: '5c1f594a-315b-4bab-9075-62b1bebed7eb',
note_key: '2026/05/19/5c1f594a-315b-4bab-9075-62b1bebed7eb/note.json'
----

v0 ノートの例:

[source,javascript]
----
note_uuid: '_f855cedd-898a-469d-87eb-ebd3e672c42f',
note_key: '2026/05/20/_f855cedd-898a-469d-87eb-ebd3e672c42f/note.json'
----

`note_key` には、元の user UUID や物理ディレクトリを含めない。

Import 後の物理保存先は、import 実行ユーザの領域に対応して次のようになる。

[source,text]
----
data/{import_user_uuid}/note/2026/05/19/5c1f594a-315b-4bab-9075-62b1bebed7eb/note.json
----

`note_uuid` 及び `note_key` は、Export / Import の前後で原則として変更しない。

==== file_uuid

`file_uuid` は、エクスポート対象ノート内の managed resource に定義されている UUID とする。

具体的には、`resource.storage.files[]` の `path` に含まれる次の `file_uuid` を使用する。

[source,text]
----
YYYY/MM/DD/file_uuid/filename
----

例:

[source,text]
----
2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/example.pdf
----

`file_uuid` は、Export / Import の前後で原則として変更しない。

Import 時に `file_uuid` を変更すると、`resource.uri`、`resource.canonicalUri`、`resource.storage.files[].path`、thumbnail、preview PDF、PageMarker、contents などの参照を一括して書き換える必要があり、バグを招きやすいためである。

==== Export package の構成

Export package は、次の構成とする。

[source,text]
----
wuwei-note-export.zip
  export-manifest.json
  note/
    YYYY/MM/DD/note_uuid/note.json
  resources/
    YYYY/MM/DD/file_uuid/
      original 又は export 用 PDF
      preview.pdf
      thumbnail.jpg
      manifest.json
      ...
----

例:

[source,text]
----
wuwei-note-export.zip
  export-manifest.json
  note/
    2026/05/19/5c1f594a-315b-4bab-9075-62b1bebed7eb/note.json
  resources/
    2026/05/05/8ed59104-779e-4907-9978-6f0f1acb8cf5/
      example.pdf
      preview.pdf
      thumbnail.jpg
      manifest.json
----

`resources/` 配下の `YYYY/MM/DD/file_uuid/` は、元の `resource.storage.files[].path` の logical base と対応する。

==== Office 文書の Export 方針

Office 文書は、Import 後に利用者が元の Office 文書を直接編集できないようにするため、`original` の Office 文書を package に同梱しない。

Office 文書の `original` が次の拡張子である場合、Export 時に OpenOffice / LibreOffice により PDF に変換し、変換後 PDF を同梱する。

[source,text]
----
.doc
.docx
.xls
.xlsx
.ppt
.pptx
.odt
.ods
.odp
----

同梱する PDF は、manifest では `role: "original"` 又は `role: "pdf-original"` として扱う。
Import 後の note JSON では、実体として参照される `original` を PDF に置き換える。

[source,json]
----
{
  "role": "original",
  "path": "resources/2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/example.pdf",
  "logicalPath": "2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/example.pdf",
  "fileName": "example.pdf",
  "mimeType": "application/pdf",
  "derivedFrom": {
    "role": "original",
    "fileName": "example.docx",
    "mimeType": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
    "exportPolicy": "office-original-replaced-by-pdf"
  }
}
----

この場合、note JSON 内の resource も、Import 後に PDF 文書として参照できるように補正する。

[source,javascript]
----
resource.kind = 'document'
resource.documentKind = 'pdf'
resource.mimeType = 'application/pdf'
resource.uri = 'YYYY/MM/DD/file_uuid/example.pdf'
resource.canonicalUri = 'YYYY/MM/DD/file_uuid/example.pdf'
resource.storage.files[role='original'].path = 'YYYY/MM/DD/file_uuid/example.pdf'
resource.storage.files[role='original'].mimeType = 'application/pdf'
resource.storage.files[role='original'].file_name = 'example.pdf'
----

元の Office 文書名は、必要に応じて次の metadata に残す。

[source,javascript]
----
resource.export = {
  originalReplacedByPdf: true,
  originalFileName: 'example.docx',
  originalMimeType: 'application/vnd.openxmlformats-officedocument.wordprocessingml.document'
}
----

==== Content resource の保存形式

Upload resource は、v2 resource 形式を保持する。

[source,javascript]
----
{
  id: '...',
  type: 'Content',
  label: '...',
  description: {...},
  resource: {
    id: '8ed59104-779e-4907-9978-6f0f1acb8cf5',
    source: 'upload',
    kind: 'document',
    documentKind: 'pdf',
    videoKind: '',
    canonicalUri: '2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/example.pdf',
    uri: '2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/example.pdf',
    title: 'example.pdf',
    mimeType: 'application/pdf',
    storage: {
      managed: true,
      files: [
        {
          role: 'original',
          area: 'upload',
          path: '2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/example.pdf',
          dir_name: 'data/{user_uuid}/upload/2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5',
          file_name: 'example.pdf',
          mimeType: 'application/pdf'
        },
        {
          role: 'thumbnail',
          area: 'upload',
          path: '2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/thumbnail.jpg',
          dir_name: 'data/{user_uuid}/upload/2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5',
          file_name: 'thumbnail.jpg',
          mimeType: 'image/jpeg'
        },
        {
          role: 'preview',
          area: 'upload',
          path: '2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/preview.pdf',
          dir_name: 'data/{user_uuid}/upload/2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5',
          file_name: 'preview.pdf',
          mimeType: 'application/pdf'
        }
      ]
    }
  }
}
----

`resource.uri`、`resource.canonicalUri`、`resource.storage.files[].path` は logical path である。

[source,text]
----
YYYY/MM/DD/file_uuid/filename
----

`resource.storage.files[].dir_name` は、サーバルート下の物理相対ディレクトリである。

[source,text]
----
data/{user_uuid}/upload/YYYY/MM/DD/file_uuid
----

Import 時には、原則として `file_uuid` と logical base は変更しない。
変更するのは、import 実行ユーザに依存する `resource.storage.files[].dir_name` である。

ただし、Office original を PDF に置き換えた resource では、`resource.uri`、`resource.canonicalUri`、`resource.storage.files[role='original'].path` のファイル名及び MIME type を PDF に更新する。

==== resource.storage.files[] の role 指定

`resource.storage.files[role='thumbnail']` は、`storage.files` 配列のうち `role` が `thumbnail` であるファイル定義を示す概念的な選択表現である。

JavaScript 実装では、`Array.prototype.find()` 又は共通 helper 関数により取得する。

[source,javascript]
----
function getResourceFileByRole(resource, role) {
  const files = resource &&
    resource.storage &&
    Array.isArray(resource.storage.files)
      ? resource.storage.files
      : [];

  return files.find(function (file) {
    return file && file.role === role;
  }) || null;
}
----

同一 resource 内では、次の role は原則としてそれぞれ 0..1 件とする。

* `original`
* `preview`
* `thumbnail`
* `pdf-preview`
* `manifest`

==== Export 処理

Export 処理は次の手順で行う。

[source,text]
----
1. 対象 note を完全 v2 形式に正規化する。
2. note_uuid 及び note_key を確認する。
3. note JSON を取得する。
4. note 内の Content node 及び note.resources[] を走査する。
5. resource.storage.files[] を持つ managed resource を抽出する。
6. resource.storage.files[] を role ごとに確認する。
7. 各 file 定義の area/path から実体ファイルを解決する。
8. Office original は OpenOffice / LibreOffice で PDF に変換し、PDF を export 対象 original として扱う。
9. PDF, image, video, audio, text, thumbnail, preview などは実体ファイルをそのまま export 作業領域へコピーする。
10. 同梱対象ファイルを resources/YYYY/MM/DD/file_uuid/ に配置する。
11. Office 置換があった場合は、export 用 note JSON の resource 参照を PDF に補正する。
12. note JSON を note/YYYY/MM/DD/note_uuid/note.json として配置する。
13. export-manifest.json を作成する。
14. 作業領域を zip 化する。
15. zip package を利用者に返す。
----

Export 作業領域は次の形式とする。

[source,text]
----
data/public/YYYY/MM/DD/package_uuid/
----

Export 完了後、この作業領域は一定期間後に削除してよい。

==== Export 時のファイル解決

Export では、`resource.storage.files[]` の各要素について、次の順序で実体ファイルを解決する。

[source,text]
----
1. file.dir_name + file.file_name
2. environment_path(file.area, export_user_uuid) + file.path
3. upload の path index:
   data/{user_uuid}/upload/_index/path/{logicalPath}.json
4. legacy content / thumbnail area:
   data/{user_uuid}/content/{path}
   data/{user_uuid}/thumbnail/{path}
----

ファイルが見つからない場合、次の扱いとする。

[source,text]
----
role が thumbnail / preview / pdf-preview:
  WARNING として manifest に記録し、Export は継続してよい。

role が original:
  ERROR RESOURCE ORIGINAL FILE NOT FOUND として Export を中止する。

Office original で PDF 変換結果が既に storage.files[] に存在する場合:
  既存の pdf-preview 又は preview PDF を original 置換 PDF として使用してよい。
----

==== Office PDF 変換

Export 時に Office original を PDF に変換する場合、変換コマンドは環境依存設定とする。

例:

[source,bash]
----
soffice --headless --convert-to pdf --outdir "$WORK_DIR" "$SOURCE_OFFICE_FILE"
----

変換に失敗した場合は、次の順序で fallback する。

[source,text]
----
1. storage.files[role='pdf-preview'] が存在すればそれを使用する。
2. storage.files[role='preview'] が PDF であればそれを使用する。
3. いずれも存在しなければ Export を中止する。
----

Office original 自体は zip package に含めない。

==== Import 処理

Import 処理は次の手順で行う。

[source,text]
----
1. zip package を受け取る。
2. data/public/YYYY/MM/DD/package_uuid/ に安全に展開する。
3. export-manifest.json を検証する。
4. note JSON を読み込む。
5. note_uuid / note_key は原則として変更しない。
6. managed resource の file_uuid も原則として変更しない。
7. resources/ 配下の同梱ファイル群を import 実行ユーザの upload 領域へコピーする。
8. resource.storage.files[].dir_name を import 実行ユーザ用に更新する。
9. Office 置換済み resource について、original が PDF を指すように note JSON を補正する。
10. upload path index を作成又は更新する。
11. audit.lastModifiedBy / audit.lastModifiedAt を import 実行ユーザと処理日時で更新する。
12. note JSON を note_key の位置に保存する。
13. 成功時は import-note.py と同じく import 済み note JSON 本体を application/json で返す。
----

Import 後の保存先は次の形式とする。

[source,text]
----
data/{import_user_uuid}/note/YYYY/MM/DD/note_uuid/note.json
data/{import_user_uuid}/upload/YYYY/MM/DD/file_uuid/
----

==== Import 時のファイル復元

Import では、manifest の `resources[].files[]` を正とし、各 file を次のように復元する。

[source,text]
----
source:
  data/public/YYYY/MM/DD/package_uuid/{file.path}

destination:
  data/{import_user_uuid}/upload/{file.logicalPath}
----

例:

[source,text]
----
source:
  data/public/2026/05/20/package_uuid/resources/2026/05/19/file_uuid/example.pdf

destination:
  data/{import_user_uuid}/upload/2026/05/19/file_uuid/example.pdf
----

Import 後、note JSON 内の `resource.storage.files[]` は次のように補正する。

[source,javascript]
----
file.area = 'upload'
file.path = file.logicalPath
file.dir_name = 'data/{import_user_uuid}/upload/YYYY/MM/DD/file_uuid'
file.file_name = filename(file.logicalPath)
----

==== path index の復元

Import では、`load-file.py` / `load-file.cgi` が logical path から実体ファイルを解決できるように、必要に応じて path index を作成する。

[source,text]
----
data/{import_user_uuid}/upload/_index/path/YYYY/MM/DD/file_uuid/filename.json
----

内容例:

[source,json]
----
{
  "logicalPath": "2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/example.pdf",
  "upload_id": "8ed59104-779e-4907-9978-6f0f1acb8cf5",
  "actual_date": "2026/05/19",
  "date": "2026/05/19",
  "file": "example.pdf",
  "manifest": "2026/05/19/8ed59104-779e-4907-9978-6f0f1acb8cf5/manifest.json"
}
----

==== 応答形式

`import-note.py` 及び `import-note.cgi` は、成功時に同じ応答形式を返す。

[source,text]
----
Content-Type: application/json; charset=UTF-8
Cache-Control: no-store

{import 済み note JSON}
----

成功時に次のような text/plain 応答を返してはならない。

[source,text]
----
OK NOTE IMPORTED
note_key=...
----

エラー時は、既存の JavaScript 側の扱いに合わせて、`ERROR ...` で始まる `text/plain` 応答を許容する。

[source,text]
----
ERROR NOTE IMPORT FAILED
ERROR NOTE ID / NOTE KEY CONFLICT
ERROR RESOURCE FILE NOT FOUND
----

==== audit の扱い

Import では、`createdBy` 及び `createdAt` は変更しない。

これらは、対象が最初に作成された履歴を示すためである。

一方、`lastModifiedBy` 及び `lastModifiedAt` は、Import 実行ユーザと Import 処理日時で上書きする。

[source,javascript]
----
audit: {
  createdBy: 'original_user_uuid',
  createdAt: '2026-05-01T10:00:00Z',
  lastModifiedBy: 'import_user_uuid',
  lastModifiedAt: '2026-05-19T12:00:00Z'
}
----

この処理は、少なくとも次の対象に適用する。

* note
* page
* node
* resource
* group
* link

Import 済みノートを再度 Export し、別ユーザが Import した場合も、`createdBy` / `createdAt` は保持し、`lastModifiedBy` / `lastModifiedAt` は新しい Import 実行ユーザと日時で上書きする。

==== 削除履歴

削除履歴は当面扱わない。

ノート、ノード、リンク、グループ、resource は UUID により識別するため、通常の Export / Import では意図しない衝突は回避できる。

削除履歴、復元履歴、変更履歴が必要になった場合は、将来拡張として `auditHistory` 又は `deletionLog` を別途定義する。

==== 実行時 URL

保存値としての `resource.uri` や `storage.files[].path` は、実体 URL ではない。
実行時に表示する URL は、`wuwei.resource` 又は共通 URL helper により生成する。

保存値:

[source,text]
----
resource.uri
resource.canonicalUri
resource.storage.files[].path
  → YYYY/MM/DD/file_uuid/filename
----

物理補助情報:

[source,text]
----
resource.storage.files[].dir_name
  → data/{user_uuid}/upload/YYYY/MM/DD/file_uuid
----

実行時 URL:

[source,text]
----
PDF / image / thumbnail / preview / managed private resource:
  load-file.py / load-file.cgi 経由

Office Viewer を使う公開可能文書:
  必要に応じて公開 URL 又は Office Viewer URL を生成

remote resource:
  resource.uri 又は resource.canonicalUri をそのまま使用
----

`load-file.py` / `load-file.cgi` は、権限確認後、`X-Accel-Redirect` により nginx の内部専用 path へ渡す。
Apache ローカル環境では、`X-Accel-Redirect` が処理されないため、CGI が直接ファイル本体を返す実装を許容する。

[source,text]
----
/_wuwei2_data/{user_uuid}/upload/YYYY/MM/DD/file_uuid/filename
----

==== セキュリティ要件

Import 時には、zip slip 攻撃を防ぐため、次の path を拒否する。

* `../` を含む path
* 絶対 path
* 空 path
* manifest に存在しない不要な file
* symbolic link
* 許可されていない root directory
* 許可されていない拡張子又は MIME type

Import は、必ず `data/public/YYYY/MM/DD/package_uuid/` の下に展開し、その後、import 実行ユーザの正式領域へコピーする。

`data/public` に展開したファイルを、Import 後の正式 resource として参照し続けてはならない。

==== 衝突処理

Import 先に同じ `note_key` 又は `note_uuid` が存在する場合は、無条件に上書きしない。

次のいずれかとして扱う。

[source,text]
----
1. 同一 note と確認できる場合
   → 利用者確認のうえで再利用又は置換する。

2. 内容が異なる場合
   → NOTE ID / NOTE KEY CONFLICT として Import を中止する。

3. 別コピーとして取り込みたい場合
   → 利用者が明示的に指定した場合に限り、新しい note_uuid / note_key を割り当てる。
----

Managed resource の `file_uuid` が衝突した場合も同様に、manifest、file size、SHA などにより同一性を確認する。
異なる内容である場合は、自動で `file_uuid` を変更せず、conflict として Import を中止する。

ただし、Office original を PDF に置き換える Export では、同じ `file_uuid` 配下の original file 名が `.docx` から `.pdf` へ変わることがある。
この場合は、manifest の `officeOriginalReplacedByPdf` 又は `derivedFrom.exportPolicy` を確認し、PDF 置換として扱う。

==== export-note.py / export-note.cgi の役割

`export-note.py` は、Export の主処理を担当する。

主な処理は次の通りである。

[source,text]
----
1. 認証済み user_id を確認する。
2. note_key 又は note_id から note JSON を取得する。
3. note JSON を v2 形式へ正規化する。
4. managed resource を抽出する。
5. resource.storage.files[] の実体ファイルを解決する。
6. Office original を PDF に変換又は既存 preview PDF に置き換える。
7. export 用 note JSON を作成する。
8. data/public/YYYY/MM/DD/package_uuid/ に note / resources / manifest を構成する。
9. zip package を生成する。
10. application/zip として返す。
----

`export-note.cgi` は、EC2 nginx で実行される CGI entry point とする。
Python 版を使用する環境では `export-note.py` を起動してよい。
Shell CGI 版を使用する環境では、同じ仕様に従って zip package を返す。

==== import-note.py / import-note.cgi の役割

`import-note.py` 及び `import-note.cgi` は、同じ意味の処理を提供する。

主な処理は次の通りである。

[source,text]
----
1. 認証済み user_id を確認する。
2. multipart/form-data の zip file を受け取る。
3. data/public/YYYY/MM/DD/package_uuid/ に安全に展開する。
4. export-manifest.json を検証する。
5. note JSON を読み込む。
6. resources/ 配下の同梱ファイルを data/{user_uuid}/upload/ へコピーする。
7. Office PDF 置換情報に従い note JSON の resource 参照を補正する。
8. resource.storage.files[].dir_name / file_name / area を補正する。
9. upload path index を作成又は更新する。
10. audit.lastModifiedBy / audit.lastModifiedAt を更新する。
11. note JSON を data/{user_uuid}/note/{note_key} に保存する。
12. 成功時は import 済み note JSON 本体を application/json で返す。
----

==== UI 表記

通常のファイル操作と区別するため、UI 表記は次とする。

[source,text]
----
Export note
Import note

ノートをエクスポート
ノートをインポート
----

`Download note` / `Upload note` という表記は、単体ファイルの download / upload と混同しやすいため使用しない。
